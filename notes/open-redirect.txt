Open Redirect


# Overview

User-controlled redirect parameters send victims to external sites without validation — common phishing enabler. Pair parameter fuzzing with encoding bypasses; confirm the final Location reaches a domain you control.

# Testing Methodology

## Overview

Identify redirect parameters, test payloads in Repeater, escalate encoding if basic values are blocked.

## Commands

See web-apps.txt (Open Redirect) and web-burp.txt for Burp/Repeater workflow.

Steps:
1. Identify redirect parameters in the application
2. Test each parameter with different payloads
3. Check if the application redirects to the malicious site
4. Try different encoding techniques if basic payloads fail
5. Verify if the application is performing proper validation


# Validation / False Positives

Redirect to /login or same-site paths is not an open redirect — confirm final Location reaches an external domain you control.
javascript: payloads require a javascript: sink, not a plain HTTP redirect — document vector separately.
Domain confusion using @ or subdomain tricks may look valid in the UI but fail in strict parsers — verify in browser, not only Burp preview.


# Prevention

1. Implement a whitelist of allowed redirect destinations
2. Use indirect references to valid destinations rather than actual URLs
3. Force all redirects to go through a page that requires the user to click a link
4. Validate URLs against a pattern of allowed domains
5. Ensure proper URL parsing and validation libraries are used


# Impact

Phishing, credential theft, session hijacking, CSRF via redirect chains, reputation damage, data exfiltration when chained with other bugs.
